# ADR: runtime bootstrap и Python-окружение

## Статус

Принято.

## Контекст

`host-migrate-kit-` разрабатывается как Debian-oriented инструмент для аудита, inventory и подготовки relocation bundle.

Во время первых практических прогонов выяснилось:
- локально проект удобно запускать в `venv`
- на чистом Debian VPS может отсутствовать `python3-venv`
- системный Python на Debian защищён PEP 668 и не предполагает свободные глобальные `pip install`
- в будущем проект должен быть переносимым и предсказуемым на разных VPS

Это означает, что модель “установим как-нибудь в system Python и запустим” слишком хрупкая.

## Решение

Основной рабочий режим проекта:
- отдельное Python virtual environment (`venv`) на каждом хосте, где запускается `host-migrate-kit-`

Нормальный ожидаемый bootstrap-путь:
1. убедиться, что на хосте есть `python3`
2. установить `python3-venv`, если его нет
3. создать `.venv`
4. установить проект внутрь окружения
5. запускать `hmk` из активированного окружения

## Что это значит practically?

### Нормальный режим

Основной поддерживаемый способ запуска:
- `python3 -m venv .venv`
- `. .venv/bin/activate`
- `pip install -e .`
- `hmk ...`

### Аварийный fallback-режим

Если на целевом или тестовом VPS нет `python3-venv`, допускается ограниченный read-only запуск без установки проекта в окружение:
- `PYTHONPATH=src python3 -m host_migrate_kit.command ...`

Этот режим считается:
- временным
- аварийным
- неполноценным

Он подходит для:
- первичного read-only аудита
- базовой диагностики хоста

Он не считается основной операционной моделью проекта.

## Почему выбран именно такой подход?

### Почему не system-wide pip install?

Потому что на Debian:
- это конфликтует с PEP 668
- повышает риск ломать системное Python-окружение
- усложняет повторяемость
- делает поведение менее предсказуемым между VPS

### Почему именно venv?

Потому что `venv` даёт:
- изоляцию проекта
- повторяемость зависимостей
- безопасность по отношению к системе
- понятный bootstrap path для нового VPS

### Почему оставляем fallback без venv?

Потому что в реальной жизни может понадобиться быстро выполнить read-only аудит на “бедном” VPS, где ещё не успели подготовить runtime.

Полный отказ от fallback-режима был бы слишком жёстким и мешал бы практической диагностике.

## Последствия решения

### Плюсы
- предсказуемый runtime
- меньше проблем с системным Python
- лучше переносимость между хостами
- понятная модель для restore guide и migration checks

### Минусы
- появляется явный bootstrap step
- на чистом VPS нужно отдельно проверять наличие `python3-venv`
- придётся документировать runtime prerequisites

## Что должно следовать из этого решения?

Проект должен содержать:
- документированный bootstrap path
- migration checks на наличие `python3`, `python3-venv` и базовых утилит
- явное разделение между нормальным runtime-режимом и аварийным fallback-режимом

## Практический вывод

Для `host-migrate-kit-` нормой считается:
- подготовить окружение перед работой
- запускать проект в собственном `venv`

Аварийный запуск без `venv` допускается только как временный read-only режим для первичной диагностики.
